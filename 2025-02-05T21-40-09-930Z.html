Cloud Computing Overview

What is cloud in computer science?

In computer science, cloud computing refers to the delivery of computing services—including servers, storage, databases, networking, software, and more—over the internet (the cloud) rather than relying on local computers or private data centers.

Key Concepts of Cloud Computing:

  1. On-Demand Access – Users can access computing resources whenever they need them, without having to buy and maintain physical infrastructure.
  2. Scalability – Cloud services can automatically scale up or down based on demand.
  3. Pay-As-You-Go Model – Users only pay for the resources they use, similar to a utility bill (e.g., electricity or water).
  4. Remote Accessibility – Cloud services can be accessed from anywhere with an internet connection.
  5. Resource Pooling – Cloud providers host and manage resources that are shared among multiple users.

Types of Cloud Computing:

  1. Public Cloud – Services provided by third-party vendors over the internet (e.g., AWS, Google Cloud, Microsoft Azure).
  2. Private Cloud – Cloud infrastructure used exclusively by a single organization.
  3. Hybrid Cloud – A combination of public and private cloud services for flexibility and security.

Cloud Service Models:

  1. Infrastructure as a Service (IaaS) – Provides virtualized computing resources like servers and storage (e.g., AWS EC2, Google Compute Engine).
  2. Platform as a Service (PaaS) – Offers development platforms with tools to build, test, and deploy applications (e.g., Google App Engine, Microsoft Azure App Service).
  3. Software as a Service (SaaS) – Delivers applications over the internet without requiring installation (e.g., Google Docs, Dropbox, Salesforce).

Advantages of Cloud Computing:

  • Reduces hardware costs and maintenance.
  • Improves collaboration and remote work.
  • Enhances security with automatic updates and backups.
  • Provides disaster recovery options.

Disadvantages of Cloud Computing:

  • Internet dependency for access.
  • Potential data privacy and security risks.
  • Possible downtime due to provider outages.

So in short cloud is just a server with shared resources?

Yes, in simple terms, the cloud is essentially a network of remote servers that provide shared computing resources over the internet. Instead of running applications or storing data on your personal computer or a local server, you access these resources from data centers managed by cloud providers like AWS, Google Cloud, or Microsoft Azure.

However, it's not just any server—cloud computing adds scalability, automation, and flexibility that traditional servers lack. Key differences include:

✅ Shared & Virtualized Resources – Instead of a single physical server, cloud providers use virtualization to divide and allocate computing power dynamically.

✅ On-Demand & Scalable – Unlike a traditional server, where you have a fixed amount of CPU, RAM, and storage, cloud resources can scale up or down automatically based on demand.

✅ Pay-Per-Use Model – You don’t own the hardware; you just rent resources as needed, reducing costs.

✅ Distributed & Redundant – Cloud data is often replicated across multiple servers and locations for better availability and disaster recovery.

So yes, at its core, the cloud is a collection of powerful, remote servers with automated management and shared resources, making it far more flexible and efficient than a single on-premise server. 🚀

If the data and computing is distributed doesn't it need a server acting as the gateway?

Yes, exactly! In cloud computing, a gateway server (or load balancer) plays a crucial role in managing access to distributed resources. This ensures that users can seamlessly interact with cloud services without worrying about where their data is physically stored or processed.

How Gateway Servers Work in Cloud Computing:

  1. Load Balancing – A gateway distributes incoming requests across multiple servers to prevent overload and ensure smooth performance.
  2. Traffic Routing – It directs users to the appropriate server or data center based on factors like location, workload, and latency.
  3. Security & Authentication – Acts as a gatekeeper, enforcing security policies such as authentication, encryption, and API rate limiting.
  4. Caching & Optimization – Some gateway servers cache frequently requested data to reduce response times and improve efficiency.

Common Types of Gateway Servers in Cloud Computing:

  • API Gateway (e.g., AWS API Gateway, Kong) – Manages API requests, rate limiting, and authentication.
  • Load Balancers (e.g., AWS Elastic Load Balancer, Nginx) – Distributes traffic across multiple instances.
  • Edge Servers/CDN (e.g., Cloudflare, AWS CloudFront) – Caches content closer to users to speed up access.

So while cloud computing is distributed, a centralized gateway server (or a system of gateway servers) manages access, security, and routing to ensure everything runs smoothly. 🚀

Is a single user's data stored on differenet servers?

Yes, a single user's data can be stored on multiple servers, depending on the cloud architecture and the service being used. This is done for redundancy, performance, and scalability.

How Data is Stored in Cloud Computing:

  1. Replication for Redundancy

    • Cloud providers often store multiple copies of your data across different servers or even different data centers to prevent data loss in case of hardware failure.
    • Example: In Amazon S3, objects are automatically replicated across multiple storage nodes.

  2. Sharding for Scalability

    • Large datasets are often split into smaller chunks (shards) and distributed across different servers. This improves performance by spreading the load.
    • Example: In distributed databases like MongoDB or Google Spanner, a large dataset is broken into smaller parts stored on different nodes.

  3. Caching for Performance

    • Frequently accessed data may be cached in different locations (e.g., edge servers in a Content Delivery Network, or CDN) to reduce latency.
    • Example: Cloudflare or AWS CloudFront caches data close to users to speed up access.

  4. Geo-Distribution for Availability

    • Cloud providers may store data in multiple geographic locations so that users in different regions get faster access.
    • Example: Google Drive automatically distributes files across multiple data centers.

Does This Mean My Data is Scattered Across Many Servers?

  • For regular users? Not necessarily. Small-scale data (e.g., your Google Docs file) might stay in one primary location with backups.
  • For large-scale applications? Definitely. Big platforms like YouTube or Netflix store and distribute data across multiple locations.

Is My Data Secure if Stored Across Multiple Servers?

Yes, because cloud providers use encryption, access control, and redundancy to protect data. However, privacy concerns depend on how the provider manages and secures these distributed copies.

So, in short: Yes, your data might be stored on multiple servers, but how it’s distributed depends on redundancy needs, performance optimizations, and cloud provider policies. 🚀

In case that the response to a request is stored on multiple servers how does the load balancer then handles it?

When a response to a request is stored on multiple servers, the load balancer handles it by using various routing and data consistency strategies to ensure efficient and accurate delivery. Here's how it works:

1. How Load Balancers Handle Distributed Data:

Load balancers do not just distribute requests; they also ensure that users receive the correct and most up-to-date data from the right server. They achieve this through:

1.1. Routing to the Nearest or Best Server

  • Geo-aware routing: If the same data exists on multiple servers worldwide, the load balancer sends the request to the nearest server to minimize latency.
  • Least connection method: Routes the request to the server with the least active connections for optimal performance.

1.2. Ensuring Data Consistency Across Servers

If data is stored on multiple servers, it’s crucial that all copies remain consistent. The load balancer works with replication and synchronization mechanisms:

  • Strong Consistency (Database-Level Sync)

    • Some cloud services use a primary (leader) database to handle writes and then sync the data with replicas (read-only nodes).
    • Load balancers always route read requests to the closest replica and write requests to the leader.

  • Eventual Consistency (Faster, but Not Always Up-to-Date)

    • In highly distributed systems like NoSQL databases (e.g., Amazon DynamoDB, Cassandra), updates propagate across servers over time.
    • Load balancers may serve slightly outdated data if immediate consistency isn’t required.

1.3. Caching for Speed

  • Load balancers may route requests to cache servers (e.g., Redis, Cloudflare CDN) to serve frequently accessed data quickly.
  • Cached responses prevent unnecessary database queries and reduce load.

2. Example Scenarios

Case 1: A Web Page Request (CDN-Based)

🔹 If a user requests a webpage (e.g., a video on YouTube), the Content Delivery Network (CDN) routes them to the closest cached server for faster delivery.

Case 2: A Database Query (Read/Write Splitting)

🔹 A user querying their order history on an e-commerce site:

  • Load balancer routes read requests to a read replica (faster, scalable).
  • Load balancer routes write requests (e.g., updating an order) to the primary database for consistency.

Case 3: Real-Time Data (Strong Consistency Required)

🔹 A banking transaction must always be accurate.

  • Load balancer always directs the request to the master database, ensuring no outdated data is shown.

3. Summary

💡 When data is stored across multiple servers, a load balancer intelligently routes requests based on:

  • Latency (geo-routing)
  • Data consistency needs (strong vs. eventual consistency)
  • Load balancing strategies (least connections, caching, CDN, database replicas)

This ensures that users always get fast, reliable, and accurate responses while optimizing cloud resources. 🚀

In this model, a single response (e.g., a file, object, or query result) is split into multiple chunks, stored across different servers, and then reassembled when needed. Here's how it works:

1. How Distributed Data Retrieval Works

When a client requests an object that is stored in multiple parts across different servers, the gateway server (or load balancer) handles retrieval using these steps:

Step 1: Request Forwarding

The gateway server receives a request for an object and determines which servers hold different chunks of the object.

Step 2: Parallel Fetching from Storage Nodes

Instead of waiting for a single large response from one server, the gateway sends multiple sub-requests in parallel to different storage nodes where parts of the object are stored.

Step 3: Reassembling the Object

Once all the chunks are retrieved, the gateway reconstructs the full object in the correct order.

Step 4: Sending the Response to the Client

After reassembling, the full response is sent back to the requesting client.

2. Where is This Used in Cloud Computing?

This approach is widely used in distributed storage and databases, including:

2.1. Object Storage (e.g., Amazon S3, Google Cloud Storage)

  • Large files (videos, backups) are split into chunks, stored across multiple servers, and retrieved in parallel.
  • Ensures high availability and quick access.

2.2. Distributed Databases (e.g., Google Bigtable, Amazon DynamoDB, Apache Cassandra)

  • Large datasets are sharded (partitioned) across multiple database nodes.
  • A query retrieves pieces of data from multiple nodes before assembling the final result.

2.3. Content Delivery Networks (CDNs) (e.g., Cloudflare, Akamai)

  • Large media files are split and cached on multiple edge servers.
  • Requests for video streaming or website assets are assembled from various locations.

2.4. Distributed File Systems (e.g., Hadoop HDFS, Ceph, GlusterFS)

  • Files are stored as blocks on different servers.
  • The system retrieves blocks in parallel and merges them before delivering the file.

✅ Scalability – Large datasets are spread across multiple servers, avoiding bottlenecks.

What is it called you said?

The process you're describing is commonly referred to as sharding, distributed storage, or parallel retrieval, depending on the context. Here are the specific terms used in different areas of cloud computing:

1. Sharding (for Databases & Storage)

  • Definition: The process of splitting large datasets or objects into smaller shards (chunks) and storing them across multiple servers.
  • Used in: Distributed databases (e.g., MongoDB, Cassandra, Amazon DynamoDB), object storage (e.g., Amazon S3, Google Cloud Storage).
  • Example: A user request for a large dataset is processed by fetching data from multiple shards in parallel.

2. Distributed File Systems (DFS)

  • Definition: A file system where data is broken into blocks and spread across multiple storage nodes.
  • Used in: Hadoop HDFS, Ceph, Google File System (GFS).
  • Example: A large file is split into blocks, stored across different servers, and retrieved in parallel.

3. Parallel Retrieval (Parallel I/O)

  • Definition: A technique where multiple storage nodes or database servers are queried at the same time to retrieve different parts of an object or dataset.
  • Used in: Big data analytics, high-performance computing (HPC).
  • Example: Fetching parts of a large file simultaneously from different storage servers to improve speed.

4. Content Delivery Network (CDN) Caching

  • Definition: Distributing chunks of large media files across multiple edge servers to reduce latency and improve speed.
  • Used in: Cloudflare, Akamai, AWS CloudFront.
  • Example: When streaming a video, different video segments are retrieved from multiple CDN nodes.

5. Erasure Coding (EC) (Data Redundancy & Fault Tolerance)

  • Definition: A technique where a file is broken into data and parity fragments and spread across multiple servers. If a part is lost, it can be reconstructed from the remaining pieces.
  • Used in: Cloud storage systems (e.g., Amazon S3, Google Cloud Storage).
  • Example: Instead of simple replication, erasure coding ensures data recovery with fewer storage overheads.

Final Answer:

The most general term for this process in cloud computing is sharding or distributed storage with parallel retrieval, but the exact term depends on the specific use case (e.g., database sharding, distributed file systems, CDN caching, or erasure coding). 🚀

Where Chunking is Used

How does the load balancer then determines which servers to fetch the data from?

A load balancer determines which servers to fetch the data from based on routing algorithms, metadata, and indexing systems that track where data chunks are stored. The exact method depends on whether the system is handling chunked data, sharded data, or replicated data.

1. How Load Balancers Locate Data

When a user requests an object (e.g., a video, file, or database record), the load balancer follows these steps:

Step 1: Look Up Metadata or Index

  • The system maintains a metadata store or index that maps object IDs or data chunks to specific storage nodes.
  • Examples:
    • Distributed file systems (HDFS, Ceph, GFS) use a NameNode or Metadata Server to store chunk locations.
    • Databases (Cassandra, DynamoDB) use consistent hashing to determine which node contains specific data.

Step 2: Choose the Best Storage Node

  • The load balancer decides which server to fetch data from based on:
    1. Location Awareness (Geo-Based Routing) – Selects the closest server to reduce latency.
    2. Load & Availability (Least-Connections, Least-Load) – Avoids overloaded servers.
    3. Cache Awareness (Edge/CDN Nodes First) – If data is cached in a CDN or in-memory cache (Redis, Memcached), it fetches from there first.

Step 3: Fetch Data in Parallel

  • If the object is chunked, the load balancer sends parallel requests to multiple storage nodes.
  • If the data is sharded, it queries multiple database nodes and merges results.

Step 4: Reassemble & Send Response

  • If data was split into chunks, the load balancer reconstructs the object before delivering it to the user.

2. Load Balancing Strategies for Data Retrieval

The load balancer can use different strategies based on system architecture:

🔹 Hash-Based Data Distribution (Sharding)

  • Uses a hash function to determine where a specific data piece is stored.
  • Example: Amazon DynamoDB, Apache Cassandra, MongoDB.

🔹 Metadata Server Lookups

  • Uses a central index to track chunk locations and fetch them efficiently.
  • Example: Hadoop HDFS, Google File System (GFS), Ceph.

🔹 Request-Based Routing (Caching/CDN)

  • First checks edge cache/CDN nodes before querying backend storage.
  • Example: Cloudflare, AWS CloudFront, Netflix CDN.

🔹 Consistent Hashing (NoSQL Databases, Caching)

  • Distributes data across storage nodes based on hashed keys, making it easy to locate.
  • Example: Amazon DynamoDB, Redis, Memcached.

3. Example Use Cases

📌 Example 1: Cloud Storage (Google Drive, Dropbox)

  • A user requests a large file.
  • The load balancer checks metadata to find where file chunks are stored.
  • It fetches multiple chunks in parallel and reassembles them before sending them to the user.

📌 Example 2: Video Streaming (Netflix, YouTube)

  • A user plays a 4K video.
  • The load balancer routes requests to the nearest CDN node.
  • If the requested chunk isn’t cached, it fetches it from backend storage nodes.

📌 Example 3: Distributed Database Query (Amazon DynamoDB, Cassandra)

  • A user queries an e-commerce database.
  • The system hashes the query key and routes it to the correct database shard.
  • If multiple shards are needed, they are queried in parallel, and the results are merged.

4. Summary

🔹 The load balancer doesn’t randomly select servers; it uses metadata, hashing, and routing logic to fetch data efficiently.
🔹 It prioritizes the nearest, least-loaded, and most cached copies of the data to optimize performance.
🔹 Parallel fetching and intelligent request routing ensure fast response times.

Explain load balancing strategies for data retrieval

Load Balancing Strategies for Data Retrieval

Load balancers use different strategies to efficiently retrieve, distribute, and process data from cloud storage, distributed databases, and web servers. The goal is to ensure high availability, low latency, and optimal performance.

1. Round Robin

📌 How It Works:

  • The load balancer distributes incoming requests sequentially across available storage nodes.
  • Each node gets a turn in a circular order.

✅ Pros:

  • Simple and easy to implement.
  • Useful when all storage nodes have equal capacity.

❌ Cons:

  • Doesn't consider server load or data location.
  • Not ideal for systems with varying resource usage.

🛠️ Use Case:

  • Small-scale distributed storage systems with uniform servers.

2. Least Connections

📌 How It Works:

  • Requests are routed to the server with the fewest active connections.
  • Helps balance workloads dynamically.

✅ Pros:

  • More efficient than Round Robin in handling varying workloads.
  • Prevents overloading busy servers.

❌ Cons:

  • Requires real-time monitoring of connections.

🛠️ Use Case:

  • Distributed databases (e.g., MySQL Cluster, PostgreSQL).
  • File storage systems with frequent concurrent access.

3. Least Response Time

📌 How It Works:

  • The request is sent to the server that has the fastest response time.
  • Uses latency monitoring to measure real-time server performance.

✅ Pros:

  • Reduces delay by selecting the best-performing server.
  • Ensures low-latency data retrieval.

❌ Cons:

  • May overload the fastest server if not combined with other strategies.

🛠️ Use Case:

  • Web services & APIs that require real-time data access.
  • Video streaming platforms like Netflix, YouTube.

4. Hash-Based (Consistent Hashing)

📌 How It Works:

  • Uses a hash function (e.g., hash(key) % num_servers) to determine which server stores a piece of data.
  • Ensures that the same request always goes to the same storage node.

✅ Pros:

  • Minimizes data movement when scaling.
  • Ensures efficient and predictable routing.

❌ Cons:

  • If a server fails, data retrieval can be disrupted.
  • Requires a distributed hash table (DHT) to keep track of changes.

🛠️ Use Case:

  • Distributed NoSQL databases (Cassandra, DynamoDB).
  • Content Delivery Networks (CDNs) to map cached files.

5. Geo-Based Routing

📌 How It Works:

  • Requests are directed to the nearest data center based on user location.
  • Reduces latency by minimizing network distance.

✅ Pros:

  • Improves user experience with faster response times.
  • Reduces bandwidth costs for global services.

❌ Cons:

  • Not useful if data is heavily replicated worldwide.
  • Requires accurate geo-DNS configuration.

🛠️ Use Case:

  • Cloud storage (Google Drive, Dropbox).
  • Streaming & CDNs (Cloudflare, AWS CloudFront).

6. Priority-Based Load Balancing

📌 How It Works:

  • Assigns priority levels to different servers.
  • Requests are sent to higher-priority nodes first.

✅ Pros:

  • Ensures high-priority resources are always available.
  • Ideal for systems with mixed hot (frequent) and cold (rare) data.

❌ Cons:

  • Low-priority nodes may remain underutilized.
  • Needs careful resource allocation.

🛠️ Use Case:

  • AI model training (allocating high-power GPUs).
  • High-availability cloud applications.

7. Cache-Aware Load Balancing

📌 How It Works:

  • First checks if the requested data is cached in-memory (Redis, Memcached) or CDN edge servers.
  • If not found, then retrieves it from storage.

✅ Pros:

  • Reduces database queries and storage I/O.
  • Improves read speeds for frequently accessed data.

❌ Cons:

  • Requires additional cache management logic.
  • Ineffective if data changes frequently.

🛠️ Use Case:

  • APIs & web apps with frequent database queries.
  • Streaming services caching popular videos.

8. Weighted Load Balancing

📌 How It Works:

  • Assigns weights to servers based on their capacity.
  • More powerful servers get more requests.

✅ Pros:

  • Ensures optimal use of high-performance resources.
  • Balances load based on CPU, RAM, network bandwidth.

❌ Cons:

  • Needs regular monitoring and tuning.

🛠️ Use Case:

  • Hybrid cloud storage (mix of on-premise & cloud).
  • Distributed analytics workloads.

Conclusion

Each load balancing strategy has its strengths and weaknesses. Modern cloud infrastructure combines multiple strategies to handle scalability, speed, and fault tolerance. 🚀

🔹 For large-scale cloud storage – Use hash-based & least response time.
🔹 For global services – Use geo-based & cache-aware.
🔹 For high-performance databases – Use least connections & weighted balancing.

  • If the video is found, it is served directly from the cache instead of fetching it from AWS.

✅ If Cached:

  3. Chunks are assembled and streamed in real-time.

📌 How Load Balancer Helps?

  • If a server is under high load, the load balancer switches to another storage node.

Step 5: Continuous Monitoring & Failover Handling

Netflix uses AI-driven load monitoring to detect:

  • Server failures
  • High traffic spikes
  • Network congestion

If a CDN node or AWS storage node fails:
✅ The load balancer instantly redirects traffic to another healthy server.
✅ This prevents downtime, ensuring the video continues streaming without interruption.

What are the mostly seen techs in cloud servers? Docker and Kuber?

Yes! Docker and Kubernetes (K8s) are among the most commonly used technologies in cloud servers, but they are part of a much broader ecosystem. Cloud servers use a variety of containerization, orchestration, virtualization, storage, and networking technologies to ensure scalability, efficiency, and high availability.

🚀 Most Commonly Seen Technologies in Cloud Servers

1️⃣ Containerization & Orchestration

These technologies allow applications to run in isolated environments for portability and efficiency.

✅ Docker – Used for creating and running containers (lightweight, portable environments for applications).
✅ Kubernetes (K8s) – Automates deployment, scaling, and management of containerized applications.
✅ Podman – A Docker alternative for managing OCI-compliant containers without a daemon.
✅ Docker Swarm – A built-in orchestration tool in Docker for managing multiple containers across clusters.

🔹 Why? Containers make cloud applications scalable, portable, and efficient.

2️⃣ Virtualization & Hypervisors

Cloud providers use virtualization to run multiple virtual machines (VMs) on a single physical server.

✅ VMware ESXi – Enterprise-grade hypervisor for virtual machines.
✅ KVM (Kernel-based Virtual Machine) – Linux-based hypervisor for creating VMs.
✅ Hyper-V – Microsoft’s virtualization technology.
✅ Proxmox VE – An open-source alternative for managing VMs and containers.

🔹 Why? Virtualization enables multi-tenant cloud environments by allowing multiple users to share the same physical infrastructure.

3️⃣ Cloud Storage & Databases

Cloud servers need highly available and scalable storage solutions.

✅ Amazon S3 (Simple Storage Service) – Object storage for cloud applications.
✅ Ceph – Distributed storage system for high availability.
✅ MinIO – Open-source alternative to Amazon S3.
✅ Google Cloud Storage – Used for handling large-scale storage.
✅ PostgreSQL & MySQL (Cloud-Optimized) – Popular relational databases used in cloud environments.
✅ MongoDB Atlas & DynamoDB – NoSQL databases for scalable cloud applications.
✅ Redis & Memcached – In-memory databases for caching.

🔹 Why? Storage solutions enable reliable, scalable, and efficient data access.

4️⃣ Networking & Load Balancing

Cloud networking technologies ensure efficient traffic distribution and communication between cloud servers.

✅ NGINX & HAProxy – Popular load balancers and reverse proxies.
✅ Traefik – Cloud-native, Kubernetes-ready load balancer.
✅ Cilium – Kubernetes-native networking with security enforcement.
✅ Istio & Linkerd – Service meshes for managing microservices networking.
✅ AWS ELB (Elastic Load Balancer) & Google Cloud Load Balancer – Managed load balancing solutions.

🔹 Why? These technologies optimize performance, security, and reliability for cloud applications.

5️⃣ Infrastructure as Code (IaC) & Automation

Managing cloud infrastructure manually is inefficient, so cloud engineers use automation tools.

✅ Terraform – Declarative IaC tool for provisioning cloud resources.
✅ Ansible – Automates configuration management.
✅ Pulumi – Infrastructure as Code using real programming languages.
✅ CloudFormation – AWS-native infrastructure automation tool.
✅ Helm – Kubernetes package manager for managing applications.

🔹 Why? IaC tools make cloud scalable, reproducible, and automated.

6️⃣ Serverless & Edge Computing

Cloud services are evolving to serverless architectures where developers don't manage the underlying infrastructure.

✅ AWS Lambda – Runs serverless functions in response to events.
✅ Google Cloud Functions – Similar to AWS Lambda for event-driven apps.
✅ Cloudflare Workers – Runs serverless applications at the edge.
✅ Knative – Kubernetes-based serverless framework.

🔹 Why? No need to manage servers! Just deploy code and let the cloud handle scaling.

7️⃣ Security & Observability

Security, monitoring, and logging are critical in cloud environments.

✅ Prometheus & Grafana – Monitoring and visualization tools.
✅ Datadog & New Relic – Cloud monitoring services.
✅ Vault (by HashiCorp) – Secure secret management.
✅ ELK Stack (Elasticsearch, Logstash, Kibana) – Centralized logging and analysis.
✅ Falco – Security runtime monitoring for containers.

🔹 Why? Observability and security are crucial to prevent downtime and attacks.
